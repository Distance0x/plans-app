# plans-app 功能缺口与迭代规划

依据：`滴答清单 与 plans-app 的功能与技术采纳研究报告.md`，并结合当前 `src/` 与 `electron/` 源码现状。

## 结论

当前 `plans-app` 的本地桌面版主链路已经成立：任务 CRUD、清单/标签、保存过滤器、浮窗、提醒与重复规则、日历、附件、番茄钟、导入导出都已经有可用实现。

现在的问题已经从“功能缺失”转为“体验精修和长期产品化”：

1. 本地版 P0/P1 已经补齐基础闭环：默认提醒、提醒操作、重复规则扩展、议程视图、附件打开、保存过滤器。
2. 后续主要是增强质量：更多测试、细节设置、统计复盘、备份回滚、重复任务投影。
3. 完全缺失：账号、同步、协作、移动端、小组件、开放 API、权限与隐私治理。

短期不建议追 TickTick 全量功能。当前更合理的结论是：本地桌面版可以进入收尾和试用阶段，服务端、多端同步、协作与开放生态全部后置。

## P0：先补产品骨架

目标：让本地桌面版具备稳定的任务组织、提醒和重复规则，避免继续在硬编码视图上堆功能。

### 1. 标签闭环

现状：`tags` 和 `task_tags` 表已经存在，但没有形成完整功能链路。

实施范围：

- 新增 tag IPC：创建、更新、删除、列表、绑定任务、解绑任务。
- 新增 `tag-store` 或合并进 `task-store`，前端缓存标签列表。
- `TaskForm` 支持选择/新建标签。
- `TaskFilters` 支持按标签过滤。
- `TaskItem` 和详情面板展示标签 chips。
- 备份导入导出继续覆盖 `tags/task_tags`，并处理重复标签名。

验收标准：

- 用户可以创建标签并绑定到任务。
- 标签能在列表、详情、过滤中一致显示。
- 删除标签不会删除任务。
- 导入备份后标签关系仍正确。

### 2. 项目/清单模型

现状：侧边栏有 Today/Recent/Inbox，但没有真正的清单或项目模型。

建议模型：

```text
lists
- id
- name
- color
- order_index
- archived_at
- created_at
- updated_at

tasks
- list_id
```

实施范围：

- 新增 `lists` 表，迁移现有任务到默认收集箱。
- 侧边栏改为：智能清单 + 用户清单。
- 新建任务时可选择清单。
- 支持按清单过滤和任务移动清单。

验收标准：

- 用户能创建多个清单。
- 任务必须属于一个清单，旧数据自动归入收集箱。
- 侧边栏数量与任务状态一致。

### 3. 重复规则升级

现状：`RecurrenceRule` 只支持 daily/weekly/monthly，UI 还没有星期选择。

短期不做完整 RFC 5545，但内部结构要向 RRULE 子集靠拢：

```ts
type RepeatFrequency = 'DAILY' | 'WEEKLY' | 'MONTHLY' | 'YEARLY';

interface RepeatRule {
  freq: RepeatFrequency;
  interval: number;
  byWeekday?: number[];
  byMonthDay?: number[];
  until?: string;
  count?: number;
  exceptions?: string[];
}
```

实施范围：

- 保持兼容旧 `frequency` 字段，读取时转换为新结构。
- 周重复支持选择周一到周日。
- 月重复支持每月某日和每月最后一天。
- 完成重复任务时生成下一实例，并继承备注、附件、标签、清单。

验收标准：

- 每周一三五、每 2 周、每月 15 日、每月最后一天可用。
- 到达截止日期或次数后不再生成。
- 旧重复任务不会崩溃。

### 4. 提醒能力补强

现状：已有 `reminders` 表和 15 秒轮询引擎，但主要是单提醒。

实施范围：

- 一个任务允许多个提醒。
- 默认提醒设置：准时、提前 5/15/30 分钟、提前 1 天。
- 通知快捷操作：完成、稍后 10 分钟、打开任务。
- 任务完成/删除时取消 pending 提醒。
- 重复任务生成下一实例时同步生成提醒。

验收标准：

- 同一任务可配置多个提醒。
- 点击通知能定位任务。
- 点击稍后提醒会创建新的 pending reminder。

## P1：把体验做成“强本地版”

目标：让核心日用体验接近成熟个人效率工具。

### 5. 全文搜索与保存过滤器

现状：搜索只查 `title/description`，且智能视图硬编码。

实施范围：

- 若继续使用 sql.js，先扩展 LIKE 到 title/description/notes。
- 如果迁移到原生 SQLite，启用 FTS5。
- 新增 `saved_filters` 表保存过滤规则。
- 支持按状态、优先级、清单、标签、日期范围、是否有提醒、是否重复组合筛选。

验收标准：

- 搜索能命中备注内容。
- 用户能保存一个自定义过滤器并显示在侧边栏。
- Today/Next 7 Days/Inbox 使用同一套过滤规则引擎。

### 6. 日历增强

现状：月/周/日视图和拖拽已有基础。

实施范围：

- 增加议程视图。
- 日历中显示重复任务未来实例投影，而不是只显示实体任务。
- 支持隐藏已完成任务。
- 支持未排期任务拖入日历。
- 增加日历设置：工作时间、周起始日、时间粒度。

验收标准：

- 用户可以从收集箱拖任务到某天某时间。
- 重复任务未来实例能在日历看到。
- 日历设置重启后保留。

### 7. 附件产品化

现状：附件是 JSON 字符串保存本地路径。

建议模型：

```text
attachments
- id
- task_id
- original_name
- stored_path
- mime_type
- size
- created_at
```

实施范围：

- 选择附件后复制到应用数据目录，避免原始文件移动导致失效。
- 任务详情展示文件名、大小、打开按钮。
- 删除任务时可选择保留或删除附件副本。
- 导出时打包附件元数据，后续再支持附件实体导出。

验收标准：

- 原始文件移动后，应用内附件仍可打开。
- 缺失附件有明确提示。

### 8. 番茄钟统计入口

现状：番茄钟本身较完整，但统计入口在侧边栏没有落地。

实施范围：

- 统计页面展示今日/本周专注时长、完成番茄数、中断数。
- 任务详情展示关联番茄记录。
- 任务列表显示实际番茄数与预估番茄数。

验收标准：

- 完成一个番茄后统计立即更新。
- 任务实际番茄数与 session 记录一致。

## P2：差异化能力

目标：做 TickTick 有、但不应早于核心骨架的能力。

### 9. 中文自然语言快速添加

实施范围：

- 先做规则解析，不接大模型。
- 支持“明天 9 点开会”“下周三下午交稿”“每周一 10 点复盘”。
- 解析结果在快速添加框下方预览，用户可一键清除。

验收标准：

- 常见中文日期时间表达可稳定落到 dueDate/dueTime/recurrenceRule。
- 原始标题可选择是否移除时间文本。

### 10. 习惯打卡

实施范围：

- 独立 `habits` 与 `habit_logs` 表。
- 支持每日/每周目标、连续天数、补签。
- 不与任务强耦合，只在 Today 中可选展示。

验收标准：

- 习惯打卡不污染任务列表。
- 连续天数和完成率正确。

### 11. 桌面便签

实施范围：

- Electron 新增 always-on-top 小窗。
- 只展示 Today 或某个 saved filter。
- 支持快速完成、打开主窗口、快速添加。

验收标准：

- 关闭主窗口后，便签可继续显示。
- 便签数据与主窗口同步刷新。

## P4：长期产品化

目标：如果未来明确引入服务端和多端同步，再做团队协作和生态。

### 14. 协作共享

实施边界：

- 只允许共享清单/项目，不做任意单任务共享。
- 权限从 owner/editor/viewer 三档开始。
- 支持指派人、评论、活动记录。

暂不做：

- 实时多人编辑。
- 复杂组织架构。
- 企业审计。

### 15. 开放 API 与集成

优先顺序：

1. URL Scheme：快速添加任务、打开任务、打开清单。
2. ICS 订阅导出：让外部日历只读查看任务。
3. 邮件转任务：给每个账号一个 inbox 邮箱。
4. REST API：只开放任务创建、查询、完成。

开放 API 必须等领域模型稳定后再做，否则会过早锁死内部设计。

## 不建议现在做的事

- 不建议现在做协作。没有账号、同步、权限模型时做协作会返工。
- 不建议现在做移动端。核心领域模型未稳定，移动端会复制技术债。
- 不建议现在接大模型 NLP。规则解析足够覆盖高频输入，成本更低。
- 不建议现在开放 API。内部模型还会变化，API 会把变化成本外部化。
- 不建议继续只美化 UI。当前最大瓶颈是产品结构和数据模型，不是视觉。

## 下一步开发切入点

优先从“标签 + 清单”开始，因为它们是智能过滤、日历筛选和后续扩展的基础边界。

建议第一个可开发任务拆成：

1. 新增 `lists` 表和 `tasks.list_id` 字段，旧任务迁移到默认收集箱。
2. 新增 tag/list IPC handlers。
3. 扩展 preload 与 `electron.d.ts`。
4. 新增前端 store。
5. 改造 `TaskForm`、`TaskFilters`、侧边栏。
6. 更新备份导入导出。
7. 跑 `npm run build` 验证。
